Status Update

February 18, 2019

Hey there,

This is just a quick site reorganization update and report on progress for Cynobroad, our main area of focus right now. Current work is now located in the home section, as that is much more important than these smaller style less important blogs, and random blogs will be here. Everything will still get dropped into the archive after the next post comes out. Exciting work is happening with Cynobroad, with the server and client now completely using a packet-like structure instead of the url scheme, which allows much better maintainability, readability, and expandability. This, along with another backend update, should allow us to bring multiple server connections and multiple channels per server into reality much easier, as well as give us a framework for images and emoji to be added to the message. As always, our testing yielded that Cynobroad uses less than 20 MB of RAM, even with the self-hosting option enabled, making our app lightweight and quick to use. On the frontend side of things, Cynobroad now has notification status indicators in place, allowing users to change from Online, Away, Busy, and Invisible modes. Functionality is currently nonexistent with this, as Cynobroad is currently incapable of sending a desktop notification or allow users to simulate being disconnected from the server without actually disconnecting, but the proof of concept is there and shows we can expand the app functionality using the packet structure. The next few updates of Cynobroad will be fixing issues with disconnection and unexpected server closure, shutting down a server safely, and moving the client code to a separate class to allow multiple client classes to be generated, similar to the server. This will provide the basis for multiple server connections. Stay tuned!

Multispatial Logistics:

Multispatial Logistics, the EVE Online based webservice for players built on ASP.NET Core MVC 2.1, began a total redo mainly because the old project broke for some reason (who knows why). It now uses a localdb SQL database to hold stargate information which means that the site will no longer return an error code because it was taking too long to call ESI several hundred times to get static data. It features a basic frontend and a placeholder for OAuth, which means fun things can be integrated such as using character skills, implants, active ship, fatigue information, location and destination data, and more to provide a better experience. Right now, the backend is about 50% of where I'd like it to be - ship database information still needs to be added, along with some related calculations regarding align times and such which would provide more accuracy towards time calculations. The frontend is practically nonexistent save for a basic form element and SSO button, so that definitely needs to be improved upon as well. I have some ideas for what I want the frontend to look like, but unfortunately no time to properly sketch them out to further polish and implement the ideas.

SDProductions Official Website

The SDProductions website has also been redesigned to feature a cleaner, more smooth browsing experience and be more concise and clear in its delivery of information. It has been changed from the previous one-page experience to a tabbed single-page no-scroll site, which I think looks cleaner. It has been modified (not using fluid grids or anything) to look decent on mobile screens and 2:3 screens, but currently has not been tested on ultrawide screens and if anything, probably breaks.

EVE ExtChat

EVE ExtChat, a side project to Multispatial Logistics, is a Win32 based tool meant for use alongside the EVE Online client. It is 100% EULA and TOS compliant, as it only reads from log files the game outputs and never interacts with the game directly. The idea and purpose of ExtChat is to provide an enhanced experience with EVE's chat system, bringing EVE's chat more inline with features expected from a modern chat platform such as Discord. Planned features include notification sounds whenever your character or an alt character's name gets mentioned in a channel, with Do Not Disturb modes available, localized intelligence alerts for enemy hostiles reported in in-game intel channels, and customizable alerts related to in-game chat. It is the first Win32 app by SDProductions to use the latest SDBrand theming consistency. However, as it is a side project, it will likely see much less attention than other ones at the moment.

September 9th, 2018

Behold, the Modulus Field Checker! This Python 3.x script will take an integer input and check if that mod set is a field under the 11 axioms of multiplication and addition. It features no exception handling whatsoever, so try not to screw up your inputs. This was created to not have to bother writing out 11 axiom checks for each of 4 modulus tables we had to create, because my fragile hands can't take writing out more than six letters at a time. Of course, my algebra teacher hasn't specifically said we could show our work this way, but I believe this adequately demonstrates my knowledge about the concept at hand and how I can apply it elsewhere. The script was verified using existing knowledge about modular operations.

July 8th, 2018

Quick update here -

After working offline tinkering about with the to-be-renamed Cynobroad chat program, I feel like it's at a point where it reaches a few requirements of mine to be published: A) It isn't entirely made up of code from StackOverflow, and B) it at least somewhat doesn't resemble a moist pile of non-working electronic waste. The GitHub repo is available here. Here's the rundown about what the program's gone through development-wise so far.

Originally, I intended it to use UDP - it's quick and easy to work with, and suits my needs as I had planned to distribute this software to friends to use on a local network, thereby not requiring a server. Heh.. yeah, unfortunately not. Although, now I can confirm all the jokes on UDP are true. I'd tell you some, but I don't think you'd get them.

This, of course, left TCP as one of my main options. From what I'd seen, TCP was a right pain to work with compared to UDP, although I did locate a few helpful sources that greatly simplified the information I needed to know. Since all I really needed was a small decentralized thing that could run anywhere, using a TCP client/server relationship seemed a little overkill, but I went with it anyway as it seemed useful and worthwhile enough to spend the time to learn how to implement it.

And that's where the program stands today. It's still a very very early stages sort of thing, with a basically non-functional UI apart from the chat system, and there's no way to distinguish who's who and who said what. Despite this, since the main difficult bit is out of the way (Networking is hard, okay?), more regular updates will be following.

That's it, see you next time.

File data within Trackr's folders
